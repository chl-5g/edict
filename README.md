⚔️ 三省三部制 · Edict

用 1300 年前的帝国制度，重新设计 AI 多 Agent 协作架构。
不是复古，而是用“分权制衡”解决多 AI 协作的失控风险。

---

🏛️ 什么是三省三部制？

三省三部制 是一套 流程安全架构，专门解决多 AI Agent 协作时的 决策黑箱、质量失控、执行偏离、责任模糊 问题。

它将人类官僚体系中最精华的 分权制衡 思想，转化为 AI 协作的 强制流程 + 独立审计 + 步骤级审批。

```
         👑 皇帝（用户）
               │
               ▼
         📜 中书省（Gemini）   ← 宏观规划
               │
               ▼
         🔍 门下省（GPT）      ← 审核封驳
               │
               ▼
         📮 尚书省（Claude）   ← 拆解调度
               │
         ┌─────┼─────┐
         ▼     ▼     ▼
      户礼部 工兵部 吏刑部   ← 三部执行（信息/构建/质量）
         │     │     │
         └─────┼─────┘
               ▼
         ⚖️ 御史台（GPT-5.4）  ← 独立审计 + 审批门禁
               │
               ▼
         👑 皇帝
```

---

🎯 解决了什么问题？

问题 三省三部制的解法
决策黑箱 强制流程：规划→审核→拆解→执行→审计，每一步有记录
质量失控 门下省“封驳”机制，在审核层拦截错误方案
执行偏离 子任务清单需皇帝过目，每步完成后审计，通过才继续
责任模糊 每个阶段有明确负责部门，审计报告可追溯
资源滥用 尚书省拆解任务+资源预估，御史台审计成本

📋 角色与职责

三省（决策与制衡）

Agent 模型 职责
中书省 Gemini 3.1 Pro 理解需求，产出《项目总体方案》（技术栈、模块划分、验收标准）
门下省 Gemini 3.1 Pro 审核方案可行性、风险、资源；不合格则 封驳 退回
尚书省 Claude Opus 4.6 拆解总体方案为子任务，分发给三部，协调进度

三部（执行）

Agent 模型 职责 输出示例
户礼部 Claude Opus 4.6 数据层 + 文档 schema.sql, swagger.yaml, README.md
工兵部 Claude Opus 4.6 开发 + 安全编码 业务代码、安全加固、Dockerfile
吏刑部 GPT-5.4 测试 + 合规 测试报告、合规意见、缺陷列表

一台（审计）

Agent 模型 职责
御史台 Gemini 3.1 Pro 收集执行日志，生成《审计奏折》，作为 审批门禁

🔁 核心流程（步骤级审批）

```
1. 中书省规划 → 产出《项目总体方案》
2. 门下省审核 → 通过 / 封驳（驳回则返回1）
3. 尚书省拆解 → 生成子任务清单
4. 皇帝审阅子任务清单 → 作出批复
   ├── 【准奏】→ 进入执行阶段
   └── 【驳回】→ 任务终止或返回中书省修改方案（返回1）
5. 执行第一个子任务（户礼部/工兵部/吏刑部）
6. 子任务完成 → 保存日志 → 同步进度给三省（中书、门下、尚书）和御史台
7. 御史台审计 → 生成《审计奏折》
8. 皇帝审阅奏折 → 作出批复
   ├── 【准奏】→ 继续下一个子任务（返回5）
   └── 【驳回】→ 任务暂停，皇帝提出修改意见 → 修改意见经门下省审核、中书省修订方案 → 返回3（重新拆解）
9. 所有子任务完成后 → 御史台生成《最终审计奏折》→ 归档
```

📖 各模块功能详解

中书省（规划中枢）

· 职责：理解皇帝（用户）旨意，产出《项目总体方案》，包含技术栈选型、模块划分、任务依赖、验收标准、资源预估。
· 输入：用户原始需求（自然语言、文档、图片等）。
· 输出：结构化的《项目总体方案》（JSON/Markdown），含任务目标、分解建议、风险提示。
· 流程位置：决策起点，产出经门下省审核。

门下省（审核封驳）

· 职责：对中书省的方案进行合法性、合理性、可行性审查。若方案存在缺陷、风险过高或与需求不符，可 封驳 退回，并附修改意见。
· 输入：中书省的《项目总体方案》。
· 输出：审核结论（“准奏”或“封驳”），封驳时附带修改建议。
· 流程位置：质量闸门，确保方案质量后再进入执行层。

尚书省（调度拆解）

· 职责：将门下省通过的总体方案拆解为可执行的子任务清单，并分发给户礼部、工兵部、吏刑部。负责协调三部资源，跟踪整体进度。
· 输入：通过的《项目总体方案》。
· 输出：《子任务清单》，包含每个子任务的责任部门、依赖关系、预估耗时、验收标准。
· 流程位置：承上启下，拆解后提交皇帝审批子任务清单。

户礼部（信息处理）

· 职责：处理所有“信息类”产出，包括数据库设计、数据接口、API文档、技术文档、用户手册等。
· 输入：尚书省分配的子任务（如“设计用户表”“生成API文档”）。
· 输出：数据模型脚本、文档文件、数据报告等。
· 流程位置：执行层信息生产部门，与其他两部并行。

工兵部（开发与安全）

· 职责：负责核心功能开发、安全编码、部署配置。在开发过程中内嵌安全措施（防注入、权限校验、加密等）。
· 输入：尚书省分配的子任务（如“实现登录接口”“配置CI/CD”）。
· 输出：业务代码、配置文件、Dockerfile、部署脚本等。
· 流程位置：执行层核心建造部门，输出可运行系统。

吏刑部（质量与合规）

· 职责：负责测试验证、代码规范检查、安全扫描、合规审计。确保工兵部产出符合质量要求，并出具测试报告。
· 输入：尚书省分配的子任务（如“编写单元测试”“执行安全扫描”）以及工兵部的产出。
· 输出：测试报告、缺陷列表、合规意见、质量评分。
· 流程位置：执行层质量闸门，测试通过后方可进入御史台审计。

御史台（独立审计）

· 职责：独立于三省三部，对每一子任务的执行日志、产出物进行全面审计。生成《审计奏折》，并作为审批门禁——只有皇帝批准，任务才能继续。
· 输入：所有部门的执行日志、产出物、测试报告。
· 输出：《审计奏折》（含问题汇总、风险评级、改进建议）及最终审计报告。
· 流程位置：贯穿执行全过程，每步审计后由皇帝决策，实现人类在环控制。

· 子任务清单必须皇帝过目：确保人类对任务分解的最终确认，防止方向错误。
· 进度同步：每个子任务完成后，三省和御史台实时掌握最新进度，决策层始终知情。
· 批复仅两种：皇帝只有“准奏”或“驳回”，驳回时必须给出修改意见，确保决策清晰。
· 修改闭环：驳回的修改意见会重新进入门下审核、中书修订流程，保证问题修正后再执行。

角色 模型 核心理由

🚀 快速开始

前置条件

· OpenClaw 已安装
· Python 3.9+
· macOS / Linux

一键安装

```bash
git clone https://github.com/cft0808/edict.git
cd edict
chmod +x install.sh && ./install.sh
```

启动

```bash
# 终端 1：数据刷新循环
bash scripts/run_loop.sh

# 终端 2：看板服务器
python3 dashboard/server.py

# 打开浏览器
open http://127.0.0.1:7891
```

详细教程请查看 Getting Started
